I've been writing TypeScript professionally for about six years. It's comfortable, productive, and the ecosystem is enormous. So when I decided to build a CLI tool for managing infrastructure deployments, the obvious choice was to reach for Node.js and Commander.js like I always do.

Instead, I chose Rust. Here's why — and what I learned along the way.

## The Motivation

The tool I was building, Terracli, needed to parse large YAML files, diff complex nested structures, and shell out to cloud provider CLIs. In Node.js, this would work fine for small inputs, but I'd already hit performance walls with similar tools when dealing with thousands of resources.

Rust promised zero-cost abstractions, no garbage collector pauses, and binaries that just work without requiring a runtime. For a CLI tool that developers run dozens of times a day, startup time matters. Node.js CLI tools typically take 200-400ms just to boot. Rust binaries start in under 5ms.

## What Surprised Me

**The compiler is your pair programmer.** Coming from TypeScript, I thought I understood what a good type system feels like. Rust's compiler takes it further. The borrow checker, which I'd heard horror stories about, turned out to be less of an obstacle and more of a teacher. Every error message comes with an explanation and usually a suggestion.

```rust
fn process_config(config: &Config) -> Result<Plan, PlanError> {
    let resources = config.resources.iter()
        .filter(|r| r.enabled)
        .map(|r| resolve_resource(r))
        .collect::<Result<Vec<_>, _>>()?;

    Ok(Plan::new(resources))
}
```

The `?` operator for error propagation is elegant. No try-catch blocks scattered everywhere, no forgotten error handling. If something can fail, the type system forces you to deal with it.

**Enums are incredibly powerful.** TypeScript's discriminated unions are useful, but Rust's enums with associated data and pattern matching are on another level. Exhaustive matching means the compiler tells you when you've forgotten to handle a case.

**Cargo is the best package manager I've used.** Coming from the npm ecosystem with its `node_modules` black hole and version resolution headaches, Cargo feels like a breath of fresh air. `cargo build`, `cargo test`, `cargo publish` — everything just works.

## What Frustrated Me

**Lifetimes are genuinely hard.** While the borrow checker became intuitive after a few weeks, explicit lifetime annotations still trip me up. When you're passing references through multiple layers of function calls, the annotations can get noisy.

**The ecosystem is smaller.** Need a library for parsing a specific file format or calling a niche API? In npm, there are probably twelve options. In Rust, there might be one, and it might be unmaintained. I ended up writing more code from scratch than I expected.

**Compile times are slow.** A clean build of Terracli takes about 45 seconds. Incremental builds are faster, but coming from the instant feedback of `tsx watch`, the wait is noticeable. I learned to write more code between compilations and rely on `rust-analyzer` for inline feedback.

## The Result

Terracli ended up being about 4,000 lines of Rust. The equivalent Node.js version would probably be 2,500 lines of TypeScript — Rust is more verbose. But the binary is 8MB, starts instantly, and handles our largest infrastructure configs (12,000+ resources) without breaking a sweat.

Would I use Rust for everything? No. For web APIs, I still reach for TypeScript and Next.js. For quick scripts, Python. But for CLI tools, performance-critical libraries, and anything that needs to be distributed as a standalone binary, Rust has earned a permanent spot in my toolkit.

## Getting Started

If you're a TypeScript developer curious about Rust, here's my advice: start with [The Rust Book](https://doc.rust-lang.org/book/), build something small but real (not just tutorials), and don't fight the borrow checker — listen to it. It's usually right.
